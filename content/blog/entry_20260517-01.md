# How I managed to argue with Claude Haiku

Everyone tells me that I'm too old-fashioned, that I should give LLMs a chance because they aree so helpful in so many situations; it's mostly people that have a sort of symbiotic relationship with the so-called AIs and find them more than useful for a myriad of tasks. I have been somewhat skeptical about usefulness of LLMs for programming tasks, even though I've seen tools that some friends of mine have built almost completely by prompting, with neglectible interventions on the generated code, and these interventions being almost always some kind of of retouch, such as UI adjustments and the like. Being skeptical doesn't mean that I'm _against_ the use of AI, or that I consider vibe coding a waste of time. My point is that LLMs are probably great at implementing "normal" code, stuff that has been coded by humans over and over again, while they lack, in my opinion, the ability of finding the solution for those quirky little problems that keep you awake because you _know_ that there is a solution, but it requires some attention to constraints and relationships between the elements that play a role in the algorythm, and also requires an analysis of the problem from a more comprehensive point of view without overlooking the details. The step that solves the problem is what I would call "_the eureka step_", by mutuating, in a more general sense, the words of an old professor I had at the university.


## The case

I was working on **whenever**, in order to enhance the internal _Lua_ interpreter: I wanted to implement a way to access some sort of _shared state_ which would be accessible to different instances of the interpreter. Initiallly I thought that it could be possible to use files for this: SSDs are quite fast, and performance wasn't a concern anyway. It's possible, for some of these instances, to run concurrently, and even using lock files as guards, concurrent access to the shared state caused deadlocks, leaving two or more _Lua_ interpreters hanging. I needed a more atomic way of handling critical sections, and this required an intervention on the **whenever** code. And, because of that, I finally decided to implement the shared state as maps that would remain in memory, which are more reliable compared to files.

What I needed at this point was a map of mutexes, which would be accessible to all of the _Lua_ threads: a thread could try to lock a mutex, identified by a string (possiblly creating and locking one if that particular mutex doesn't exist yet), and other threads would wait when trying to lock the mutex identified by the same string. Easy. A little difficulty was, that I would have liked the possibility for the thread to try to lock a mutex for a certain time, and in case the lock didn't succeed, to stop trying and go on, possibly failing in a gentle way.

I already had a solution in mind, which was to implement a `Hashmap` that associated owned `String`s to simple `Mutex` objects: _Rust_ would handle the atomicity of every operation, and present to _Lua_ a simple API to access this facility.

The problem seemed to me interesting and isolated enough to give AI a try: I searched some hints on how to write a prompt and decided to use _Claude_ as offered (for free) by GitHub.


## First attempt

I tried to write a prompt in _Markdown_, in plain English. Every suggestion I found said almost the same things:

* be clear while describing the expected result
* describe the functions that the module should provide
* include examples for as many cases as possible.

This is the prompt I ended up redacting:

```markdown
# Named Mutex

Write a Rust module implementing a set of named mutexes. The named mutexes 
must be shared across all modules in the program and available to all the 
threads in the program. The named mutexes should be accessible via two 
functions:

1. `namedmutex_lock(name: &str, timeout: Option<usize>) -> bool`
2. `namedmutex_release(name: &str) -> bool`

The `namedmutex_lock` function tries to acquire and lock the mutex that 
has the name provided in the `name` variable if it is already present in 
the set, and if not it creates a new mutex with the given name and locks 
it immediately. If a `Some(msecs)` value is provided for the `timeout` 
parameter, the function only waits for at most `msecs` milliseconds to 
acquire the lock, and returns `false` if after that period it couldn't 
acquire the lock; if `timeout` is `None`, then the function waits forever 
to acquire the lock. If the function acquires the lock successfully, it 
returns `true`.

The `namedmutex_release` releases the locked mutex whose name is provided 
in the `name` parameter, and returns true when it is released; if a mutex 
with the specified name was not found in the set, it returns `false`.

Examples:

* `if namedmutex_lock("Mux01", None) { println!("locked!"); sleep(Duration::from_millis(500)); let  _ = namedmutex_release("Mux01"); }`
  This will wait forever for a mutex named "Mux01", once locked it prints 
  something, waits for half a second, then it releases the lock.
* `if !namedmutex_lock("Mux01", 1000) { println!("could not lock the mutex"); }`
  This will wait at most a second to acquire the mutex, and if this was 
  not possible it writes a message.
  
The mutexes should be accessible by several threads at once.

Clarity in the code is more appreciated than efficiency.
```

I thought it was clear enough and followed the guidelines I found.

This is the code that Claude provided:

```rust
//! A module providing named mutexes that can be shared across all threads in the program.
//!
//! This module implements a global set of named mutexes that can be locked and released
//! by any thread in the program.

use std::collections::HashMap;
use std::sync::{Arc, Mutex, MutexGuard, TryLockError};
use std::time::Duration;

/// A wrapper around a mutex that can be identified by name
struct NamedMutexData {
    mutex: Arc<Mutex<()>>,
}

/// Global storage for all named mutexes
static NAMED_MUTEXES: std::sync::OnceLock<Mutex<HashMap<String, Arc<Mutex<()>>>>> =
    std::sync::OnceLock::new();

/// Initializes the global named mutex storage
fn get_mutex_map() -> &'static Mutex<HashMap<String, Arc<Mutex<()>>>> {
    NAMED_MUTEXES.get_or_init(|| Mutex::new(HashMap::new()))
}

/// Attempts to acquire and lock a named mutex.
///
/// If a mutex with the given `name` doesn't exist, it creates one and locks it immediately.
/// If a mutex with the given `name` exists, it attempts to lock it.
///
/// # Arguments
///
/// * `name` - The name of the mutex to lock
/// * `timeout` - Maximum time to wait for the lock in milliseconds.
///   - `None`: Wait indefinitely
///   - `Some(msecs)`: Wait at most `msecs` milliseconds
///
/// # Returns
///
/// Returns `true` if the mutex was successfully locked, `false` if the timeout was exceeded.
///
/// # Examples
///
/// '''ignore
/// if namedmutex_lock("Mux01", None) {
///     println!("locked!");
///     std::thread::sleep(std::time::Duration::from_millis(500));
///     let _ = namedmutex_release("Mux01");
/// }
/// '''
///
/// '''ignore
/// if !namedmutex_lock("Mux01", Some(1000)) {
///     println!("could not lock the mutex");
/// }
/// '''
pub fn namedmutex_lock(name: &str, timeout: Option<usize>) -> bool {
    // Get or create the named mutex
    let mutex_arc = {
        let mut mutex_map = get_mutex_map().lock().unwrap();
        mutex_map
            .entry(name.to_string())
            .or_insert_with(|| Arc::new(Mutex::new(())))
            .clone()
    };

    // Try to lock the mutex
    let result = if let Some(msecs) = timeout {
        // Attempt to lock with timeout
        let duration = Duration::from_millis(msecs as u64);
        match mutex_arc.try_lock_for(duration) {
            Ok(_) => {
                // Store the guard to keep the lock alive
                // We need to store this in thread-local storage
                store_guard(name.to_string(), mutex_arc);
                true
            }
            Err(TryLockError::Timeout) => false,
            Err(TryLockError::Poisoned(_)) => {
                // Handle poisoned mutex by recovering
                match mutex_arc.lock() {
                    Ok(_) => {
                        store_guard(name.to_string(), mutex_arc);
                        true
                    }
                    Err(_) => false,
                }
            }
        }
    } else {
        // Wait indefinitely for the lock
        match mutex_arc.lock() {
            Ok(_) => {
                store_guard(name.to_string(), mutex_arc);
                true
            }
            Err(_) => false,
        }
    };

    result
}

/// Releases a previously locked named mutex.
///
/// # Arguments
///
/// * `name` - The name of the mutex to release
///
/// # Returns
///
/// Returns `true` if the mutex was successfully released, `false` if a mutex
/// with the specified name was not found or was not locked by the current thread.
///
/// # Examples
///
/// '''ignore
/// if namedmutex_lock("Mux01", None) {
///     // ... do some work ...
///     namedmutex_release("Mux01");
/// }
/// '''
pub fn namedmutex_release(name: &str) -> bool {
    clear_guard(name)
}

/// Thread-local storage for mutex guards to keep locks alive
thread_local! {
    static MUTEX_GUARDS: std::cell::RefCell<HashMap<String, Arc<Mutex<()>>>> =
        std::cell::RefCell::new(HashMap::new());
}

/// Stores the mutex arc to maintain the lock in this thread
fn store_guard(name: String, mutex_arc: Arc<Mutex<()>>) {
    MUTEX_GUARDS.with(|guards| {
        guards.borrow_mut().insert(name, mutex_arc);
    });
}

/// Clears the guard for a named mutex, releasing the lock
fn clear_guard(name: &str) -> bool {
    MUTEX_GUARDS.with(|guards| {
        guards.borrow_mut().remove(name).is_some()
    })
}
```

## Further attempts

Actually, I thought that Claude wouldn't have invented the `try_lock_for` function just out of the blue, and that it should have existed somewhere. I investigated a little, to find out that the [_parking_lot_](https://crates.io/crates/parking_lot) crate offers it. I read the documentation, and saw that this implementation of the synchronization primitives could help me in several places in **whenever** and, considering that in fact **whenever** doesn't really deal with many contentions of shared resources, it could even be more efficient than the one found `std::sync` library.

Just to begin (in the end I actually converted all synchronization in **whenever** to _parking_lot_), i modified the last lines of the prompt in this way:


```rust
use parking_lot::Mutex;
use std::collections::HashMap;
use std::sync::Arc;
use std::time::Duration;

/// Global storage for named mutexes
/// Arc allows shared ownership across threads
/// Mutex protects the HashMap from concurrent modifications
static NAMED_MUTEXES: parking_lot::once_cell::sync::Lazy<Mutex<HashMap<String, Arc<Mutex<()>>>>> =
    parking_lot::once_cell::sync::Lazy::new(|| Mutex::new(HashMap::new()));

/// Attempts to acquire and lock a named mutex.
///
/// If a mutex with the given name doesn't exist, it creates one and locks it immediately.
/// If a mutex with the given name already exists, it attempts to lock it.
///
/// # Arguments
/// * `name` - The name of the mutex to lock
/// * `timeout` - Optional timeout in milliseconds. If `None`, waits indefinitely.
///
/// # Returns
/// `true` if the lock was acquired successfully, `false` if the timeout expired
/// before acquiring the lock.
///
/// # Examples
/// '''ignore
/// // Wait forever for the lock
/// if namedmutex_lock("Mux01", None) {
///     println!("locked!");
/// }
///
/// // Wait at most 1000ms for the lock
/// if !namedmutex_lock("Mux02", Some(1000)) {
///     println!("could not lock the mutex");
/// }
/// '''
pub fn namedmutex_lock(name: &str, timeout: Option<usize>) -> bool {
    // Get or create the mutex for this name
    let mutex_to_lock = {
        let mut mutexes = NAMED_MUTEXES.lock();
        mutexes
            .entry(name.to_string())
            .or_insert_with(|| Arc::new(Mutex::new(())))
            .clone()
    };

    // Try to acquire the lock with optional timeout
    match timeout {
        None => {
            // Wait indefinitely
            let _guard = mutex_to_lock.lock();
            true
        }
        Some(msecs) => {
            // Wait for specified duration
            let duration = Duration::from_millis(msecs as u64);
            match mutex_to_lock.try_lock_for(duration) {
                Some(_guard) => true,
                None => false,
            }
        }
    }
}

/// Releases a locked named mutex.
///
/// This function does not actually unlock the mutex (since we don't store the guard).
/// Instead, it verifies that a mutex with the given name exists.
/// The actual unlocking happens when the guard from `namedmutex_lock` is dropped.
///
/// # Arguments
/// * `name` - The name of the mutex to release
///
/// # Returns
/// `true` if a mutex with the given name exists, `false` otherwise.
///
/// # Examples
/// '''ignore
/// if namedmutex_lock("Mux01", None) {
///     // Do some work
///     let _ = namedmutex_release("Mux01");
/// }
/// '''
pub fn namedmutex_release(name: &str) -> bool {
    let mutexes = NAMED_MUTEXES.lock();
    mutexes.contains_key(name)
}
```

## Conclusion

I must admit that I was not completely dissatisfied by these sessions: at the end of the day I had come across the _parking_lot_ crate, which turned out to be useful even to make my code more readable, compared to the `std::sync` library. And I also discovered some interesting patterns in _Rust_ which were more "elegant" than the ones I used throughout my code. However, the AI generated code didn't directly solve my problem. There are several possible reasons for this: maybe the questions I pose are too difficult, or that I bumped into a language that this AI doesn't know that much. And maybe (and most probably) it's me, I might just be not so good at "_vibe coding_", which is possibly a form of coding that needs some kind of proficiency. All in all, for now I think that I will still have to think and write code by myself.

In the future, I might be tempted to try AI interaction again for something more mechanical, I cannot think about it now but in the past I bumped into coding tasks that consisted in a lot of copy-and-paste plus search-and-replace activity, which in my opinion AI might be able to cope with, and in a less error-prone way.

One thing, however, made me smile at the end of my session with Claude. I spent around four hours trying to convince it to deliver a solution that compiled and passed the tests. After I gave up, I noticed a warning on the Copilot page that said that I had consumed more than 80% of my (free) monthly AI resources. In one night. For just mockery and lies.

Ok, it is a free tool, I'm fine. But... If it were a job interview with Claude, I wouldn't hire it.
